# Generative UI & AI Integration Plan

This document outlines the roadmap to make `ct-design-infra` fully capable of powering Generative UI and AI-driven development. It serves as context for LLM agents when tackling design system tasks.

## 🟢 Current Strengths (Ready for GenUI)

1. **W3C DTCG Tokens:** The tokens defined in `packages/tokens/src/tokens.json` use the W3C Design Tokens Community Group format (`$value`, `$type`). This standardization is perfect for RAG and AI tool consumption.
2. **Semantic Web Components:** The components are built using Lit (e.g., `<ct-button>`, `<ct-accordion>`) and expose simple, string-based attributes. LLMs are highly proficient at generating standard HTML strings with attributes, making them streamable directly to the DOM without complex wrappers. Components with repeatable child content (accordion panels, and future tabs/lists) use a parent+child element pair rather than a JSON array prop, keeping every attribute on every element a plain string/boolean — see `ui/accordion`.
3. **Accessibility Data:** The `wcag-data/` directory provides excellent structured data on accessibility requirements, which can be fed to agents to enforce compliance during generation.
4. **Zod Schema per Component:** Every component ships a `<name>.schema.ts` mirroring its Lit properties (`ButtonSchema`, `AccordionSchema`/`AccordionItemSchema`, ...) — the strict boundary for constraining AI-generated props. Not yet exported as a standalone consumable package (see Action Item 1).
5. **Machine-Readable Component Manifest:** `packages/core/registry.json` is generated by `scripts/build-registry.mjs` from the Custom Elements Manifest (`dist/custom-elements.json`) as part of `pnpm build:core` — no agent or contributor hand-maintains it. It maps each tag (e.g. `ct-accordion`, `ct-accordion-item`) to its semantic description (from the component's own JSDoc), its schema module/export, and its attribute list, and flags any custom element that doesn't yet have a matching schema. Gitignored build output, not a deliverable in a diff.

## 🔴 Missing Groundwork (Action Items)

The following capabilities need to be implemented to enable safe and constrained Generative UI (e.g., via Vercel AI SDK or MCP).

### 1. Export Zod Schemas for External Consumption
**Problem:** Per-component Zod schemas exist (see Current Strengths above), but they're only reachable by importing each component's `.schema.ts` file directly — there's no single package/utility an external AI framework backend can import to get every schema at once.
**Action:**
- Export all component schemas from a single entry point (e.g. re-export from `packages/core/src/index.ts` or a dedicated `schemas.ts`), or generate standard JSON Schema output alongside `registry.json` so non-TypeScript consumers (an MCP server, a different language's AI SDK) don't need to evaluate TypeScript to get the schema shape.

### 2. "Few-Shot" Example Corpora — thin, needs backfilling
**Problem:** `ai-examples/` exists, but only `accordion.html` has a realistic composed example; `button.html` is still the scaffold's one-line placeholder (`<ct-button label="Generated Label Example">`), never backfilled after Button shipped.
**Action:**
- Write a real composed example for Button (and every component going forward — the [New Component Addition Checklist](component-addition-checklist.md) already calls for this at port time, it just wasn't enforced for the first component).

### 3. Setup Model Context Protocol (MCP) Server
**Problem:** To bring this knowledge directly into the developer's IDE (Cursor, Claude Desktop), the data needs to be served.
**Action:**
- Build a lightweight MCP server package in the monorepo that exposes the Design Tokens, `registry.json`, Schemas, and `ai-examples/` as queryable tools/resources for local agents.

## 📋 Component Addition Process

To ensure all new components are built with these GenUI capabilities from day one, follow the [New Component Addition Checklist](component-addition-checklist.md) (condensed how-to version: `.claude/skills/add-component/SKILL.md`). When asking an AI to add a component, reference this checklist so it generates the Lit component, the Zod schema, and the few-shot example together — `registry.json` then picks the component up automatically on the next `pnpm build:core`, nothing to update by hand.

To port several components at once with multiple agents running in parallel, see `docs/parallel-porting.md` and the tracking file `PORTING_STATUS.json`.
